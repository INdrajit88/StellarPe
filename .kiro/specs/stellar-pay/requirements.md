# Requirements Document

## Introduction

StellarPay is a production-ready, mobile-first payment web application inspired by UPI (Unified Payments Interface), powered by the Stellar blockchain testnet. The platform supports three user roles — User, Merchant, and Admin — each with dedicated dashboards and workflows. Users can create Stellar-backed wallets, send and receive payments, save contacts, and scan QR codes to pay merchants. Merchants can generate QR codes, accept payments, and view earnings analytics. Admins can monitor platform activity, manage users, and view engagement metrics.

The system is built with Next.js (App Router) + Tailwind CSS on the frontend, Node.js + Express (or Next.js API routes) on the backend, PostgreSQL with Prisma ORM for data persistence, and the Stellar SDK for blockchain interactions.

## Glossary

- **System**: The StellarPay web application as a whole.
- **Wallet**: A Stellar keypair (public key + AES-256-GCM encrypted secret key) held exclusively on the backend and associated with a registered user account. Users never receive, store, or interact with the secret key directly.
- **User**: A registered individual who holds a Wallet, can send/receive payments, manage contacts, and scan QR codes.
- **Merchant**: A registered business account that can generate QR codes, accept payments, and view earnings analytics.
- **Admin**: A privileged operator who can monitor platform statistics and manage User and Merchant accounts.
- **Stellar_SDK**: The official JavaScript Stellar SDK used to interact with the Stellar testnet.
- **Horizon_API**: The Stellar testnet Horizon server used to submit transactions and query account state.
- **Friendbot**: The Stellar testnet faucet service used to fund newly created accounts with test XLM.
- **Keypair**: A Stellar public/secret key pair generated by the Stellar_SDK.
- **Stellar_Address**: The public key of a Stellar account, used as the on-chain payment address.
- **Username**: A unique human-readable identifier chosen by a User or Merchant at registration, mapped to a Stellar_Address.
- **Contact**: A saved entry consisting of a display name and a Stellar_Address or Username, stored per User.
- **QR_Code**: A machine-readable image encoding a Stellar_Address and optional payment amount, generated for Merchant payment requests.
- **Transaction**: A Stellar payment operation recorded on the testnet and mirrored in the platform database.
- **Transaction_History**: A paginated, filterable list of past Transactions associated with a Wallet.
- **JWT**: A signed JSON Web Token used for stateless session authentication.
- **Auth_Provider**: The authentication subsystem (JWT-based or Clerk/Auth.js) that issues and validates session tokens.
- **Encryption_Service**: The backend service responsible for encrypting and decrypting Wallet secret keys at rest using AES-256-GCM. The plaintext secret key is never transmitted to or accessible by the client.
- **Transaction_PIN**: A numeric PIN (minimum 4 digits) set by the User at registration or profile setup, used to authorize payment and transaction operations. Functionally equivalent to a UPI MPIN. The Transaction_PIN is stored as a salted bcrypt hash and is never stored or transmitted in plaintext.
- **PIN_Service**: The backend service responsible for hashing, storing, and verifying the Transaction_PIN.
- **Custodial_Wallet**: A wallet model in which the backend generates, stores, and operates on the Stellar Keypair entirely server-side. Users do not hold or manage private keys.
- **Dashboard**: The role-specific main screen presented to a User, Merchant, or Admin after authentication.
- **Bottom_Nav**: The mobile bottom navigation bar present on all authenticated User and Merchant screens.
- **Prisma**: The ORM layer used to interact with the PostgreSQL database.
- **Environment_Config**: The set of environment variables (`.env`) required for application operation.

## Requirements

### Requirement 1: User Registration and Authentication

**User Story:** As a visitor, I want to register an account and log in securely, so that I can access my wallet and payment features.

#### Acceptance Criteria

1. THE Auth_Provider SHALL accept a registration request containing a unique username, email address, password, and role (User or Merchant).
2. WHEN a registration request is received with a duplicate username or email, THE Auth_Provider SHALL return a descriptive error indicating which field is duplicated.
3. WHEN a valid registration request is received, THE Auth_Provider SHALL store the user record in the database and automatically trigger Wallet creation for the new account.
4. WHEN valid credentials are submitted at login, THE Auth_Provider SHALL return a signed JWT with an expiry of no more than 24 hours.
5. WHEN an invalid credential combination is submitted at login, THE Auth_Provider SHALL return a 401 error with a generic "invalid credentials" message without revealing which field is incorrect.
6. WHILE a JWT is expired or absent, THE System SHALL reject all authenticated API requests with a 401 response.
7. IF a registration request is missing any required field, THEN THE Auth_Provider SHALL return a 400 error listing all missing fields.

---

### Requirement 2: Custodial Wallet Creation and Key Management

**User Story:** As a new User or Merchant, I want a Stellar wallet to be automatically created and managed for me on the backend, so that I can send and receive payments immediately after registration without needing a browser extension or handling private keys.

#### Acceptance Criteria

1. WHEN a new account is successfully registered, THE System SHALL generate a Stellar Keypair exclusively on the backend using the Stellar_SDK; the secret key SHALL never be transmitted to the client at any point.
2. WHEN a Keypair is generated, THE System SHALL fund the new Stellar account by submitting the public key to Friendbot on the Stellar testnet.
3. THE Encryption_Service SHALL encrypt the Wallet secret key using AES-256-GCM before persisting it in the database.
4. THE System SHALL store only the AES-256-GCM encrypted secret key and the plaintext public key (Stellar_Address) in the database; no other representation of the secret key SHALL be stored.
5. WHEN a User requests their Wallet details, THE System SHALL return the Stellar_Address and current XLM balance retrieved from the Horizon_API.
6. IF the Friendbot funding request fails, THEN THE System SHALL retry the request up to 3 times before returning a wallet creation error to the caller.
7. THE System SHALL never expose the plaintext secret key in any API response, client-side code, or application log.
8. THE Custodial_Wallet model SHALL require no browser wallet extension; THE System SHALL be fully operable on mobile browsers without any client-side extension or plugin.

---

### Requirement 3: Send Payment

**User Story:** As a User, I want to send a payment to another user or merchant by username or Stellar address after confirming with my PIN, so that I can transfer funds quickly and securely from any mobile browser without needing a wallet extension.

#### Acceptance Criteria

1. WHEN a User submits a send-payment request with a recipient identifier, amount, and optional memo, THE System SHALL resolve the identifier to a Stellar_Address if a Username is provided.
2. WHEN the recipient identifier cannot be resolved to a Stellar_Address, THE System SHALL return a descriptive error before submitting any transaction.
3. WHEN a valid send-payment request is received, THE System SHALL require the User to provide their Transaction_PIN before proceeding; THE PIN_Service SHALL verify the submitted PIN against the stored hash before any signing occurs.
4. IF the submitted Transaction_PIN does not match the stored hash, THEN THE System SHALL reject the payment request with an "incorrect PIN" error and SHALL NOT proceed to transaction signing.
5. WHEN the Transaction_PIN is verified, THE System SHALL decrypt the sender's Wallet secret key using the Encryption_Service on the backend and sign the Stellar payment operation via the Stellar_SDK entirely server-side.
6. WHEN a Stellar payment operation is successfully submitted to the Horizon_API, THE System SHALL record the Transaction in the database with status "completed", timestamp, sender Stellar_Address, recipient Stellar_Address, amount, and memo.
7. WHEN a Stellar payment operation is rejected by the Horizon_API, THE System SHALL record the Transaction in the database with status "failed" and the error reason, and return a descriptive error to the caller.
8. IF the sender's XLM balance is insufficient to cover the payment amount plus the Stellar minimum reserve (1 XLM), THEN THE System SHALL return an insufficient-balance error before submitting any transaction.
9. THE System SHALL ensure the plaintext secret key is held in server memory only for the duration of the transaction signing operation and is not persisted to any log or storage.
10. WHEN a payment is completed, THE System SHALL return the Transaction details including the Stellar transaction ID to the caller within 10 seconds.

### Requirement 4: PIN-Based Transaction Authorization

**User Story:** As a User, I want to authorize every payment with a numeric PIN (similar to a UPI MPIN), so that my funds are protected even if my session token is compromised.

#### Acceptance Criteria

1. WHEN a new User completes registration, THE System SHALL prompt the User to set a Transaction_PIN of 4 to 6 digits before the Wallet becomes usable for sending payments.
2. THE PIN_Service SHALL hash the Transaction_PIN using bcrypt with a minimum cost factor of 12 before persisting it; the plaintext PIN SHALL NOT be stored or logged at any point.
3. WHEN a User submits any payment or transaction-signing request, THE System SHALL require the Transaction_PIN to be included in the request payload; requests without a PIN SHALL be rejected with a 400 error.
4. WHEN a Transaction_PIN is submitted with a payment request, THE PIN_Service SHALL verify the PIN against the stored bcrypt hash before the Encryption_Service decrypts the Wallet secret key.
5. IF a User submits an incorrect Transaction_PIN 5 consecutive times within a 15-minute window, THEN THE System SHALL temporarily lock payment authorization for that account for 15 minutes and return a descriptive lockout message.
6. WHEN a User requests a PIN reset, THE System SHALL require re-authentication (valid session JWT and a verified one-time code sent to the registered email) before allowing the Transaction_PIN to be changed.
7. WHEN a User successfully changes their Transaction_PIN, THE System SHALL invalidate all active sessions for that User and require re-login.
8. THE System SHALL accept Transaction_PIN submission from a standard numeric input on mobile browsers without requiring any browser extension or external application.

---

### Requirement 5: Receive Payment and Real-Time Updates

**User Story:** As a User or Merchant, I want to be notified when I receive a payment, so that I can confirm funds have arrived without manually refreshing.

#### Acceptance Criteria

1. WHEN a payment targeting a User's or Merchant's Stellar_Address is confirmed on the Horizon_API, THE System SHALL record the inbound Transaction in the database.
2. THE System SHALL stream payment events from the Horizon_API using the Stellar_SDK streaming interface for all registered Stellar_Addresses.
3. WHEN an inbound Transaction is recorded, THE System SHALL push a real-time notification to the recipient's active browser session using WebSockets or Server-Sent Events.
4. WHEN a User or Merchant Dashboard is loaded, THE System SHALL display the current XLM balance fetched from the Horizon_API, refreshed within the last 30 seconds.
5. IF the Horizon_API streaming connection is interrupted, THEN THE System SHALL attempt to reconnect with exponential backoff up to a maximum interval of 30 seconds.

---

### Requirement 6: Contact Management

**User Story:** As a User, I want to save contacts with names and addresses, so that I can send payments to frequent recipients quickly without re-entering their details.

#### Acceptance Criteria

1. WHEN a User submits a create-contact request with a display name and a Stellar_Address or Username, THE System SHALL validate that the address or username exists before saving the Contact.
2. WHEN a valid create-contact request is received, THE System SHALL store the Contact linked to the requesting User's account.
3. WHEN a User requests their contact list, THE System SHALL return all saved Contacts for that User ordered alphabetically by display name.
4. WHEN a User submits an update-contact request, THE System SHALL update the specified Contact's display name or address in the database.
5. WHEN a User submits a delete-contact request, THE System SHALL remove the specified Contact from the database.
6. IF a User attempts to save a duplicate Stellar_Address or Username as a Contact, THEN THE System SHALL return a descriptive error indicating the duplicate.

---

### Requirement 7: QR Code Generation and Payment

**User Story:** As a Merchant, I want to generate QR codes for payment requests, so that Users can scan and pay me without manually entering my address.

#### Acceptance Criteria

1. WHEN a Merchant requests a static QR_Code, THE System SHALL encode the Merchant's Stellar_Address into a QR_Code image.
2. WHEN a Merchant requests a dynamic QR_Code with a specific amount and optional description, THE System SHALL encode the Merchant's Stellar_Address, amount, and description into a QR_Code image.
3. THE System SHALL generate QR_Code images in PNG format at a minimum resolution of 256×256 pixels.
4. WHEN a User scans a QR_Code using the in-app scanner, THE System SHALL parse the encoded Stellar_Address and pre-populate the payment form with the address and amount if present.
5. WHEN a scanned QR_Code contains a malformed or unresolvable Stellar_Address, THE System SHALL display a descriptive error to the User without initiating a payment.
6. THE QR_Code parser SHALL validate that any decoded Stellar_Address is a valid 56-character Stellar public key before pre-populating the payment form.

---

### Requirement 8: Transaction History

**User Story:** As a User or Merchant, I want to view my transaction history with filters, so that I can track my payment activity and reconcile balances.

#### Acceptance Criteria

1. WHEN a User or Merchant requests their Transaction_History, THE System SHALL return a paginated list of Transactions associated with their Wallet, defaulting to 20 records per page.
2. THE System SHALL support filtering Transaction_History by date range (start date and end date).
3. THE System SHALL support filtering Transaction_History by transaction direction (sent or received).
4. THE System SHALL support filtering Transaction_History by transaction status (completed or failed).
5. WHEN a Transaction_History request includes multiple filters, THE System SHALL apply all filters conjunctively and return only matching records.
6. WHEN a Transaction_History request is made, THE System SHALL return each Transaction record including: transaction ID, Stellar transaction ID, timestamp, sender address, recipient address, amount, memo, and status.
7. THE System SHALL return Transaction_History results within 2 seconds for result sets of up to 1000 records.

---

### Requirement 9: Pay by Username

**User Story:** As a User, I want to send a payment using a recipient's username instead of their Stellar address, so that I can transact with human-readable identifiers.

#### Acceptance Criteria

1. THE System SHALL maintain a unique mapping between each registered Username and its corresponding Stellar_Address.
2. WHEN a send-payment request includes a Username as the recipient identifier, THE System SHALL resolve the Username to its mapped Stellar_Address before constructing the payment operation.
3. WHEN a Username resolution is requested for a username that does not exist in the database, THE System SHALL return a "username not found" error to the caller.
4. THE System SHALL resolve a Username to a Stellar_Address within 500 milliseconds under normal database load.
5. WHEN a User searches for a recipient by typing a partial Username in the payment form, THE System SHALL return matching Username suggestions from the database (up to 10 results) to support autocomplete.

---

### Requirement 10: User Dashboard

**User Story:** As a User, I want a clean mobile dashboard showing my balance, recent transactions, and quick-pay actions, so that I can manage my finances at a glance.

#### Acceptance Criteria

1. WHEN a User loads their Dashboard, THE System SHALL display the current XLM balance, the 5 most recent Transactions, and quick-pay action buttons (Send, Scan QR, Pay by Username).
2. THE Dashboard SHALL render correctly on viewport widths from 320px to 1440px without horizontal scrolling.
3. THE Bottom_Nav SHALL be present on all authenticated User screens and SHALL provide navigation links to: Dashboard, Send, Contacts, History, and Profile.
4. WHEN a User taps the Send quick-pay action, THE System SHALL navigate to the send-payment form with the recipient field focused.
5. WHEN a User taps the Scan QR action, THE System SHALL activate the device camera for QR code scanning.

---

### Requirement 11: Merchant Dashboard

**User Story:** As a Merchant, I want a dashboard showing my QR code, total earnings, and recent transactions, so that I can monitor my business payments efficiently.

#### Acceptance Criteria

1. WHEN a Merchant loads their Dashboard, THE System SHALL display the Merchant's static QR_Code, total lifetime earnings in XLM, today's transaction count, and the 10 most recent inbound Transactions.
2. THE Merchant Dashboard SHALL include a "Generate Dynamic QR" action that opens a form accepting amount and description fields.
3. THE Bottom_Nav SHALL be present on all authenticated Merchant screens and SHALL provide navigation links to: Dashboard, QR Codes, Transactions, Analytics, and Profile.
4. WHEN a Merchant navigates to the Analytics section, THE System SHALL display daily transaction volume (XLM), transaction count per day, and total earnings aggregated over the last 30 days.
5. THE Merchant Analytics charts SHALL render within 3 seconds of navigation for datasets of up to 10,000 transactions.

---

### Requirement 12: Admin Dashboard

**User Story:** As an Admin, I want a platform-level dashboard showing user counts, transaction volumes, and activity metrics, so that I can monitor and manage the platform health.

#### Acceptance Criteria

1. WHEN an Admin loads the Dashboard, THE System SHALL display: total registered User count, total registered Merchant count, total Transaction count, total XLM volume transacted, and a count of failed transactions in the last 24 hours.
2. THE Admin Dashboard SHALL display a paginated list of all registered Users and Merchants, with 25 records per page.
3. WHEN an Admin searches by username or email in the user management list, THE System SHALL return matching records within 1 second.
4. WHEN an Admin deactivates a User or Merchant account, THE System SHALL set the account status to "inactive" and THE Auth_Provider SHALL reject all subsequent login attempts for that account.
5. WHEN an Admin reactivates a previously deactivated account, THE System SHALL set the account status to "active" and the account SHALL be able to log in normally.
6. THE Admin Dashboard SHALL restrict access so that ONLY accounts with the Admin role can load it; any non-Admin request SHALL receive a 403 response.

---

### Requirement 13: Security and Fraud Considerations

**User Story:** As the platform operator, I want basic security controls in place, so that user funds and data are protected from common attack vectors.

#### Acceptance Criteria

1. THE System SHALL rate-limit authentication endpoints to a maximum of 10 requests per IP address per minute, returning a 429 response when the limit is exceeded.
2. THE System SHALL rate-limit payment submission endpoints to a maximum of 20 requests per authenticated User per minute.
3. THE Encryption_Service SHALL derive the AES-256-GCM encryption key from a master secret stored exclusively in the Environment_Config and never stored in the database.
4. THE System SHALL validate and sanitize all user-supplied inputs on the server side before processing or persisting them.
5. WHEN a User's account has 5 consecutive failed login attempts, THE System SHALL lock the account for 15 minutes and return a descriptive lockout message.
6. THE System SHALL enforce HTTPS for all client-server communication in production deployments.
7. THE System SHALL include CSRF protection on all state-mutating API endpoints.
8. THE System SHALL operate fully on mobile browsers without requiring any browser wallet extension, plugin, or installed application; all cryptographic operations SHALL be performed exclusively on the backend.

---

### Requirement 14: Environment Configuration

**User Story:** As a developer, I want all secrets and environment-specific values managed via environment variables, so that the application can be deployed securely across environments.

#### Acceptance Criteria

1. THE System SHALL load all environment-specific configuration (database URL, JWT secret, Stellar network passphrase, encryption master key, Horizon API URL) from environment variables at startup.
2. THE System SHALL provide a documented `.env.example` file listing all required environment variable keys without their values.
3. IF a required environment variable is absent at startup, THEN THE System SHALL log a descriptive error identifying the missing variable and terminate the process rather than running in a misconfigured state.
4. THE System SHALL default the Stellar network to the testnet Horizon endpoint (`https://horizon-testnet.stellar.org`) when no override is provided in the Environment_Config.

---

### Requirement 15: Project Structure and Code Quality

**User Story:** As a developer, I want a clean, modular project structure with reusable components and clear separation of concerns, so that the codebase is maintainable and scalable.

#### Acceptance Criteria

1. THE System SHALL organize the frontend code into feature-based directories under `src/app/` following Next.js App Router conventions, with reusable UI components under `src/components/`.
2. THE System SHALL organize backend API logic into route handlers or Express controllers, with database access encapsulated in a dedicated Prisma service layer under `src/lib/` or `services/`.
3. THE System SHALL define all Prisma database schema models in a single `schema.prisma` file, with separate model definitions for User, Wallet, Transaction, Contact, and MerchantProfile.
4. THE System SHALL include inline code comments for all non-trivial business logic, cryptographic operations, and Stellar SDK interactions.
5. THE System SHALL use TypeScript throughout the codebase with strict mode enabled.
